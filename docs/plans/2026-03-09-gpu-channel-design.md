# GPU Channel Architecture — Drain-and-Batch Encoding

**Date:** 2026-03-09
**Issue:** #67 (follow-up)
**Status:** Design approved
**Prerequisite:** `WgpuCompressor` from #67 (already merged)

## Problem

The `WgpuCompressor` wraps `GpuBlockCompressor` in a `std::sync::Mutex`, serializing
all GPU work. Each 4096×4096 tile requires 6 separate GPU round-trips (base + 5 mip
levels), each acquiring the Mutex independently. When multiple `spawn_blocking` threads
encode tiles simultaneously, they pile up waiting on the Mutex — creating a convoy
effect that starves throughput at DSF boundaries.

The `block_compression` library is *designed* for batching — `add_compression_task()`
accumulates tasks and `compress()` dispatches all in one compute pass — but our code
calls it one-task-at-a-time.

## Solution: Approach 2 — Dedicated GPU Channel with Drain-and-Batch Worker

Replace the `Mutex<GpuBlockCompressor>` with an `mpsc` channel feeding a single
dedicated GPU worker task. The worker uses a drain-and-batch pattern to collect
multiple compression requests and dispatch them in minimal GPU passes.

### Architecture

```
┌─────────────────────┐     ┌─────────────────────┐
│  BuildAndCacheDds   │     │  BuildAndCacheDds   │
│  (spawn_blocking)   │     │  (spawn_blocking)   │
│                     │     │                     │
│  1. Assemble image  │     │  1. Assemble image  │
│  2. Gen mipmaps     │     │  2. Gen mipmaps     │
│  3. Send to GPU ──► │     │  3. Send to GPU ──► │
│  4. Await oneshot   │     │  4. Await oneshot   │
└────────┬────────────┘     └────────┬────────────┘
         │                           │
         ▼                           ▼
    ┌────────────────────────────────────┐
    │     bounded mpsc channel (32)      │
    └────────────────┬───────────────────┘
                     │
                     ▼
    ┌────────────────────────────────────┐
    │       GPU Worker Task (single)     │
    │                                    │
    │  loop {                            │
    │    batch = recv() + try_recv()     │
    │    for each request:               │
    │      upload texture(s)             │
    │      add_compression_task() × mips │
    │    compress(&mut pass) // ONE pass │
    │    submit → poll → readback        │
    │    send results via oneshot        │
    │  }                                 │
    └────────────────────────────────────┘
```

### Key Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Channel type | `tokio::sync::mpsc` bounded(32) | Backpressure when GPU saturated; async-compatible |
| Batch size | MAX_BATCH = 4 tiles | VRAM budget ~256MB safe; 4×6=24 tasks per pass |
| Response | `tokio::sync::oneshot` per request | Caller blocks on result; natural error propagation |
| Trait integration | Implement `BlockCompressor` | Drop-in replacement; no changes to DdsEncoder or task pipeline |
| Mipmap generation | Stays CPU-side (caller) | Already in spawn_blocking; ~5ms; not bottleneck |
| Worker lifecycle | Spawned at startup, dies on channel drop | Clean shutdown via RAII |

### Message Type

```rust
struct GpuEncodeRequest {
    mipmaps: Vec<RgbaImage>,     // Pre-generated by caller (CPU work)
    format: DdsFormat,
    response: oneshot::Sender<Result<Vec<u8>, DdsError>>,
}
```

### Drain-and-Batch Pattern

```rust
loop {
    // Block for first item
    let first = rx.recv().await?;
    let mut batch = vec![first];

    // Drain up to MAX_BATCH-1 more without blocking
    while batch.len() < MAX_BATCH {
        match rx.try_recv() {
            Ok(req) => batch.push(req),
            Err(_) => break,
        }
    }

    // Process entire batch in one GPU pass
    process_batch(&device, &queue, &mut compressor, batch);
}
```

### Design for Approach 3 (Future)

The channel interface is **identical** for Approach 3. Only worker internals change:
- Approach 2: batch → one compute pass → one submit → poll → readback all
- Approach 3: pipeline individual tiles — submit A → upload B → readback A → submit B → upload C → etc.

The `GpuEncodeRequest` / `oneshot` pattern works for both.

### Performance Impact

| Metric | Current (Mutex) | Approach 2 (Channel) |
|--------|-----------------|----------------------|
| GPU passes per tile | 6 (one per mip) | 1 (all mips batched) |
| GPU passes for 4 concurrent tiles | 24 | 1 |
| Mutex contention | N threads competing | None (single worker) |
| CPU↔GPU round-trips | 6 per tile | 1 per batch |
| Backpressure | None (threads pile up) | Bounded channel (32) |
